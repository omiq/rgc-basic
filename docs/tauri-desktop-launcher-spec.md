# Tauri desktop launcher for RGC-BASIC web export

**Audience:** Maintainers adding "download as desktop app" alongside the existing
**Web Export ZIP** in the online IDE. Goal: shipping `.app` (macOS), `.exe`
(Windows), and `.AppImage` / binary (Linux) that double-click to run a user's
RGC-BASIC program in a native window — no terminal, no browser, no `python3 -m
http.server`.

## Why Tauri (vs. Go+browser, Electron, Bun-compile)

| Option | Binary size | Ships browser? | Local files? | Notes |
|---|---|---|---|---|
| **Tauri** | 3–10 MB | No (uses system webview) | Native via `tauri://` asset protocol | Built for embedded static apps. Rust toolchain. |
| Go + `embed` + system browser | 5–8 MB | No (opens user's default) | Localhost server | Simplest; runs as browser tab, not "app". |
| Webview (zserge / wry standalone) | 4–10 MB | No | Localhost or custom protocol | Manual plumbing. |
| Electron | 80–150 MB | Yes (Chromium) | Yes | Heavy. Overkill. |
| Bun `--compile` | 50–90 MB | Uses default browser | Bundle + `Bun.serve` | Same UX as Go but fatter. |

Tauri wins on: small binary, true desktop window, asset protocol bypasses
file:// CORS without a localhost server, established CI bundlers per OS.

Trade-off: Linux uses **WebKitGTK** (must be present on host or bundled in
AppImage). macOS uses **WKWebView** (built in). Windows uses **WebView2**
(ships on Win10 21H2+ and all Win11; older Win10 needs the bootstrapper, which
Tauri's installer handles).

## Runtime requirements (verified)

`make basic-wasm-raylib` flags (`Makefile`):

- `WASM=1`, `USE_GLFW=3`, `FULL_ES2=1`, `ALLOW_MEMORY_GROWTH=1`
- `ASYNCIFY=1`, `FORCE_FILESYSTEM=1`, `NO_EXIT_RUNTIME=1`
- **No `USE_PTHREADS`** → no `SharedArrayBuffer` requirement → **no COOP/COEP
  headers needed**.

What this means for the launcher:

- WebGL2 context — system webview must support WebGL2. macOS WKWebView ✅,
  Windows WebView2 (Chromium-based) ✅, WebKitGTK 2.36+ ✅.
- No cross-origin isolation needed.
- WASM must be served with `Content-Type: application/wasm`. Tauri's asset
  protocol does this from the file extension.
- `fetch()` against `tauri://` works for `HTTPFETCH` of bundled-relative URLs;
  remote `https://` calls work normally subject to the remote's CORS.

## Architecture

```
┌──────────────────────────────────────────────────────────┐
│  rgc-basic-app (per-OS native binary)                    │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Tauri shell (Rust)                                │  │
│  │   - registers `rgc://` custom protocol             │  │
│  │   - serves embedded resources/* with WASM MIME     │  │
│  │   - opens window pointing at rgc://localhost/      │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  System webview (WKWebView / WebView2 / WebKitGTK) │  │
│  │   - loads index.html                               │  │
│  │   - basic-raylib.js / .wasm                        │  │
│  │   - program.bas + project-files.json + assets      │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

User flow: download `rgc-basic-myprogram-mac-arm64.dmg` (or `.msi` /
`.AppImage`) → double-click → window opens → BASIC program runs.

## Pack pipeline (IDE integration)

`desktop-launcher/scripts/pack-export.sh`:

```bash
#!/usr/bin/env bash
# Usage: pack-export.sh <export-zip> <out-dir> <target-triple>
# Produces a per-OS installer/binary at <out-dir>/.
set -euo pipefail

EXPORT_ZIP="$1"
OUT_DIR="$2"
TARGET="$3"   # e.g. aarch64-apple-darwin, x86_64-pc-windows-msvc, x86_64-unknown-linux-gnu

WORK="$(mktemp -d)"
trap 'rm -rf "$WORK"' EXIT

# 1. Stage launcher template
cp -R desktop-launcher/. "$WORK/"

# 2. Drop user's web export into resources/
unzip -q "$EXPORT_ZIP" -d "$WORK/resources/"
# Existing zip contains: index.html, program.bas, project-files.json,
#                        rgc-basic/basic-raylib.js, rgc-basic/basic-raylib.wasm
# Flatten if needed so index.html sits at resources/ root.

# 3. Build
cd "$WORK"
cargo tauri build --target "$TARGET" --bundles "$(bundle_for_target "$TARGET")"

# 4. Collect artifacts
cp -R "src-tauri/target/$TARGET/release/bundle/." "$OUT_DIR/"
```

`bundle_for_target`:

| Target triple | `--bundles` arg | Artifact |
|---|---|---|
| `aarch64-apple-darwin` | `app,dmg` | `RGC-BASIC.app` + `.dmg` |
| `x86_64-apple-darwin` | `app,dmg` | as above |
| `x86_64-pc-windows-msvc` | `nsis,msi` | `.exe` installer + `.msi` |
| `x86_64-unknown-linux-gnu` | `appimage,deb` | `.AppImage` + `.deb` |

The IDE pipeline can pre-build the **launcher binary with empty `resources/`**
once per Tauri/template version, then for each user export only re-bundle
(skip the Rust compile by using `cargo tauri build --no-bundle` cached state +
manual asset replacement). Realistically: **~30–90 s per OS per export** on
first cut; optimisation is a follow-up.

## Code signing & notarisation — **deferred**

**Decision for v1: ship unsigned. User does one-time bypass per OS.** Paid
certs revisited only if support burden grows. Per-OS first-run instructions
live in [`docs/desktop-launcher-first-run.md`](desktop-launcher-first-run.md);
short version embedded in the download README.

Cost vs friction:

| OS | Unsigned friction | Signed cost |
|---|---|---|
| macOS | Right-click → Open (or `xattr -dr com.apple.quarantine` on Sequoia) | Apple Developer ID ~$99/yr + notarytool |
| Windows | SmartScreen "More info → Run anyway" once per user per binary | EV cert ~$200–500/yr |
| Linux | `chmod +x` for AppImage; `dpkg -i` for `.deb` | None |

When promoting to signed (future):

- macOS: `codesign` + `xcrun notarytool submit` after `cargo tauri build`.
  Tauri config supports `APPLE_CERTIFICATE`, `APPLE_ID`, `APPLE_PASSWORD`,
  `APPLE_TEAM_ID` as CI secrets.
- Windows: `signtool sign` via Tauri's `bundle.windows.signCommand`.
- Linux: optional GPG sign for `.deb` repos.

## Bundle-size estimate

Per-OS shipped artifact (raylib-emscripten WASM + JS ~2.5 MB; Tauri shell
~3–5 MB; user assets variable):

- macOS `.dmg`: **6–10 MB**
- Windows `.msi`: **5–8 MB** (WebView2 bootstrapper adds ~2 MB if offline
  installer chosen)
- Linux `.AppImage`: **45–80 MB** (bundles WebKitGTK; `.deb` is 5–8 MB but
  requires the user to have `libwebkit2gtk-4.1` installed)

Linux AppImage size is the only ugly number — comes from the bundled
WebKitGTK. Acceptable trade for "double-click to run".

## Known issues / open questions

1. **Emscripten `locateFile`** — verify `basic-raylib.js` loads `.wasm`
   correctly under `tauri://`. If not, patch `index.html` to set
   `Module.locateFile`. Test before shipping.
2. **`HTTPFETCH` to `file://`-equivalents** — Tauri's `tauri://` is treated as
   a secure context, so `fetch` works. Confirm `HTTPFETCH("./asset.png", ...)`
   for relative resources resolves; if user code uses absolute `https://`
   only, no issue.
3. **Gamepad / `PEEK(56320 + code)` keyboard matrix** — WKWebView and
   WebView2 expose Gamepad API and standard `KeyboardEvent`. Should match
   browser behaviour. Smoke-test `examples/gfx_jiffy_game_demo.bas` on each
   OS.
4. **Audio (`SOUND` / MOD playback)** — WebAudio works in all three
   webviews. WebKitGTK has historically lagged on AudioWorklet — verify with
   a MOD-loading sample.
5. **Window resize / `#OPTION columns`** — Tauri window is independent of
   canvas size; the existing rAF copy loop handles this. Verify
   `#OPTION border` padding renders correctly.
6. **File drag-drop** — out of scope v1. Future: enable Tauri's drag-drop
   handler to write into `Module.FS` for runtime asset injection.
7. **Stop / Pause buttons** — `web/index.html` already exposes Run/Pause/Stop;
   keep the same UI inside the Tauri window. No native menu bar in v1.
8. **Auto-update** — out of scope v1. Each program export is a frozen
   snapshot, so updating the launcher means re-exporting from the IDE.

### Architecture revision: external assets, not embedded

POC (current state) bakes `resources/` into the binary via
`tauri::generate_context!`. Per-export rebuild needs Rust + ~30 s compile —
incompatible with browser-only IDE.

**Switch the launcher to read assets from disk**, sibling to the binary,
via a custom URI scheme handler in Rust:

```rust
fn main() {
    tauri::Builder::default()
        .register_uri_scheme_protocol("app", |_ctx, request| {
            let path = request.uri().path().trim_start_matches('/');
            let exe = std::env::current_exe().unwrap();
            // mac: .app/Contents/MacOS/binary → ../Resources/app/
            // win/linux: ./app/ next to the binary
            let base = if cfg!(target_os = "macos") {
                exe.parent().unwrap().join("../Resources/app")
            } else {
                exe.parent().unwrap().join("app")
            };
            let file = base.join(path);
            let mime = mime_for_path(&file);
            let body = std::fs::read(&file).unwrap_or_default();
            tauri::http::Response::builder()
                .header("Content-Type", mime)
                .body(body).unwrap()
        })
        .invoke_handler(tauri::generate_handler![toggle_fullscreen])
        .run(tauri::generate_context!())
        .expect("launcher failed");
}
```

`tauri.conf.json` window URL becomes `app://localhost/index.html`. Now the
launcher binary is a **frozen template** — drop user files into the sibling
folder, no recompile.

### Distribution form: portable archives, not installers

`.dmg` / `.AppImage` / NSIS-`.exe` are not modifiable in-browser:

- `.dmg`: HFS+ image, no JS-side tooling.
- `.AppImage`: squashfs, no JS-side tooling.
- NSIS `.exe`: embedded archive, requires recompile.

Ship as **portable archives** the IDE can unzip + repack with JSZip:

```
launcher-mac-arm64.zip        →  RGC-BASIC.app/
                                   Contents/MacOS/rgc-basic-launcher
                                   Contents/Resources/app/      ← inject here
launcher-mac-x64.zip          →  same
launcher-win-x64.zip          →  rgc-basic-launcher.exe
                                   app/                          ← inject here
launcher-linux-x64.tar.gz     →  rgc-basic-launcher
                                   app/                          ← inject here
```

UX trade vs `.dmg` / `.AppImage`: user double-clicks the zip → Finder /
Explorer / file-manager extracts → user runs the app inside. Same number
of clicks as a dmg drag-install. AppImage portable still works on Linux
without install (`chmod +x` then run).

### 8bitworkshop integration steps

**1. Add launcher fetcher** (`src/ide/desktop-launcher.ts`, new file)

```ts
const LAUNCHER_BASE = "https://cdn.retrogamecoders.com/launcher/v0.1.0";
type Platform = "mac-arm64" | "mac-x64" | "win-x64" | "linux-x64";
const FILENAMES: Record<Platform, string> = {
  "mac-arm64": "launcher-mac-arm64.zip",
  "mac-x64":   "launcher-mac-x64.zip",
  "win-x64":   "launcher-win-x64.zip",
  "linux-x64": "launcher-linux-x64.tar.gz",
};
async function fetchLauncherTemplate(p: Platform): Promise<Uint8Array> {
  const r = await fetch(`${LAUNCHER_BASE}/${FILENAMES[p]}`);
  if (!r.ok) throw new Error(`launcher ${p}: ${r.status}`);
  return new Uint8Array(await r.arrayBuffer());
}
```

**2. Inject user files into template** (extend `shareexport.ts`)

New function `_downloadRgcBasicDesktopZipFile(e, platform)` parallels
existing `_downloadRgcBasicWebZipFile`:

```ts
export async function _downloadRgcBasicDesktopZipFile(e, platform: Platform) {
  // Reuse current web-export builder to produce the resources payload
  const webZip = await buildRgcBasicWebPayload();   // refactor existing fn

  // Fetch + parse launcher template
  const tplBytes = await fetchLauncherTemplate(platform);
  const tplZip = await JSZip.loadAsync(tplBytes);

  // Locate the inject path per platform
  const injectRoot =
    platform.startsWith("mac")
      ? findAppBundleRoot(tplZip) + "/Contents/Resources/app"
      : "app";

  // Copy each web-export file into the launcher's app/ folder
  webZip.forEach((path, file) => {
    if (file.dir) return;
    tplZip.file(`${injectRoot}/${path}`, file.async("uint8array"));
  });

  // Save
  const out = await tplZip.generateAsync({ type: "blob" });
  saveAs(out, `${projectName}-${platform}.zip`);
}
```

For `linux-x64` (tar.gz), use `pako` (already a dep) for gzip + a tiny tar
parser/writer (~200 lines, or grab `tar-stream` browser build).

### Pipeline summary

```
8bitworkshop (browser)
  ├─ existing: build webZip in memory
  ├─ NEW: fetch launcher-{platform}.zip from CDN
  ├─ NEW: load both with JSZip
  ├─ NEW: copy webZip files into launcher's app/ subdir
  ├─ NEW: generate combined zip
  └─ saveAs("project-mac-arm64.zip")

User
  └─ unzip → double-click .app / .exe / linux binary → game runs
```

Zero server. Zero Rust on IDE side. Launcher templates rebuilt only when
the Rust shell changes (rare — most updates are to the embedded
`basic-raylib.{js,wasm}` which lives in the per-export injected payload,
not the template).
